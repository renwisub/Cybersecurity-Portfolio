# Multi-Stage Attack Investigation: Brute-Force → Lateral Movement → Kerberoasting

## Задача

Симулированный сценарий (на основе BOTSv1 dataset паттернов) в Splunk: обнаружена цепочка событий,
указывающая на многоэтапную атаку — brute-force на внешний IP, успешный логон, последующее
латеральное перемещение по внутренним хостам и попытка Kerberoasting через запросы TGS-билетов
со слабым RC4-шифрованием (Ticket_Encryption_Type=0x17).

Цель — восстановить полный таймлайн атаки одним SPL-пайплайном и оценить, успел ли злоумышленник
закрепиться в инфраструктуре.

## Что искал

- `EventCode=4625` — неудачные попытки логона (brute-force)
- `EventCode=4624, Logon_Type=3` — сетевые логоны на несколько хостов (lateral movement)
- `EventCode=4769, Ticket_Encryption_Type=0x17` — запросы Kerberos-тикетов со слабым шифрованием (Kerberoasting)

## Процесс

1. Собрал все три типа событий в один base search через `OR`
2. Через `eval` + `case()` разметил каждое событие понятным `event_type`, с учётом приоритета
   условий (более специфичное — `Logon_Type=3` — проверяется раньше общего `EventCode=4624`)
3. Использовал `transaction user maxspan=1h`, чтобы Splunk сам склеил связанные события в единый
   инцидент по пользователю в разумном временном окне — достаточном, чтобы захватить всю цепочку
   (~35-40 минут), но не настолько большом, чтобы склеивать случайные несвязанные события за сутки
4. Отсортировал по `_time` для читаемого таймлайна

```spl
index=main
(EventCode=4625 OR EventCode=4624 OR (EventCode=4624 AND Logon_Type=3) OR (EventCode=4769 AND Ticket_Encryption_Type=0x17))
| eval event_type=case(
    EventCode=4625, "Brute_Force_Fail",
    EventCode=4624 AND Logon_Type=3, "Lateral_Movement",
    EventCode=4624, "Successful_Logon",
    EventCode=4769 AND Ticket_Encryption_Type="0x17", "Kerberoasting"
)
| transaction user maxspan=1h
| sort _time
| table _time, event_type, user, src_ip, dest, EventCode
```

## Вывод (Incident Report)

**Summary:** Зафиксирована успешная атака типа Kerberoasting, инициированная с внешнего IP-адреса
[X]. Злоумышленник использовал учётную запись [Y] для массового запроса сервисных билетов (TGS)
со слабым типом шифрования (RC4). Атака привела к расшифровке пароля и успешному
несанкционированному входу. Признаков закрепления (persistence) не обнаружено.

**Timeline:**
- `14:00` — первые аномальные TGS-запросы с шифрованием 0x17 (RC4)
- `14:05` — пик активности: >100 запросов EventCode=4769 с IP [X], похоже на автоматизированный инструмент (напр. Rubeus)
- `14:15` — успешный вход (EventCode=4624) под скомпрометированной учётной записью [Y]

**Severity:** High — учётные данные скомпрометированы и использованы для интерактивного входа,
но не Critical, поскольку реагирование опередило создание бэкдоров, повышение привилегий или
lateral movement.

**IOCs:**
- Внешний IP: `[X]`
- Скомпрометированная учётная запись: `[Y]`
- Ticket_Encryption_Type: `0x17` (RC4)

**Recommended Actions:**
1. Изоляция: блокировка IP [X] на firewall, разрыв активных сессий
2. Сброс паролей для [Y] и всех сервисных аккаунтов, чьи билеты запрашивались в окне атаки
3. Ретроспективная проверка (EventCode 4720, 4624 Logon_Type=3/10, 4732) на предмет закрепления — не обнаружено
4. Долгосрочно: отказ от RC4 в пользу AES-256 через GPO; переход критичных сервисных аккаунтов на gMSA (автоматическая ротация 120-символьных паролей раз в 30 дней)

## Инструменты

Splunk / SPL — `transaction`, `case()`, `eval`
